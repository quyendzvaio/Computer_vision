Phân tích khó khăn & điểm chưa tối ưu

Nhìn thẳng vào các vấn đề tiềm ẩn trước khi design chi tiết

🔴 Vấn đề 1: GIL + QThread — Inference có thực sự chạy song song?

Hiểu nhầm phổ biến: QThread trong Python vẫn bị GIL (Global Interpreter Lock) ràng buộc.

Nếu cả CAM1 (YOLOv8) và CAM2 (YOLO-Pose) đều chạy inference trong QThread riêng:

  • ONNX Runtime giải phóng GIL khi chạy C++ kernel → inference thực sự song song
  • Nhưng Python code xử lý (crop, keypoint analysis, overlay drawing) vẫn chạy tuần tự
  • Nếu dùng GPU và cả 2 model cùng chiếm VRAM → có thể OOM trên edge device
Giải pháp: Cân nhắc CAM1 inference trong process riêng (giữ shared memory cũ) nếu CPU/GPU không đủ mạnh. Hoặc dùng GPU model nhỏ hơn (YOLOv8n vs YOLOv8s).

🔴 Vấn đề 2: 2-stage PPE (Pose → crop → classify) — Tiềm ẩn latency cao

Mỗi frame CAM2 phải chạy:

YOLO-Pose (full frame) → N keypoint sets → Với mỗi person: → crop vùng đầu → classifier nhẹ → helmet? → crop vùng thân → classifier nhẹ → vest? → crop vùng chân → classifier nhẹ → boot? Hậu xử lý + overlay → gửi đi

Vấn đề:

  • YOLO-Pose chậm hơn YOLOv8 detect (~1.5-2x) do phải predict thêm keypoint head
  • 3 classifier nhỏ cho 1 người, nếu có 5 người = 15 lần classify/frame
  • Keypoints noise → crop sai → classifier kết quả tệ
  • Tổng latency có thể 70-120ms/frame trên CPU, chưa kể capture + overlay
Giải pháp: Skip frame strategy — chỉ chạy classifier 1 lần mỗi N frame. Hoặc dùng model detection thường cho PPE (YOLOv8 multi-class: person/helmet/vest/boot) rẻ hơn. Hoặc dùng YOLO-Pose rồi chỉ classify khi keypoint confidence > threshold.

🟡 Vấn đề 3: PyQt mất accessibility so với Web Dashboard

  • PyQt là desktop app → chỉ xem được trên máy chạy code (hoặc qua VNC/X11 forwarding)
  • Không thể xem từ mobile, tablet, laptop khác trong mạng
  • Không có REST API cho bên thứ ba tích hợp
  • Không có history page, admin page sẵn (hiện tại dashboard web có)
Giải pháp: Hybrid — PyQt cho real-time monitoring chính. Giữ FastAPI + Web dashboard nhẹ cho viewing/mobile, dùng MediaMTX RTSP để cấp video stream thay vì base64 WS. Giữ REST API cho history/admin.

🟡 Vấn đề 4: MediaMTX quản lý lifecycle + encoding overhead

  • MediaMTX là process riêng — PyQt phải khởi tạo, monitor, restart khi crash
  • Frame gửi vào MediaMTX cần encode H264/H265 → tiêu tốn CPU/GPU (có thể 5-10ms/frame)
  • Nếu mục tiêu là xem trên PyQt là chính, encode RTSP là chi phí phụ không cần thiết
  • Thêm 1 failure point vào system
Giải pháp: MediaMTX optional — bật khi cần xem từ xa. Mặc định chỉ render PyQt local. Hoặc dùng GStreamer pipeline nhúng thay vì process riêng.

🟡 Vấn đề 5: Object tracking cho ROI exit detection

Simple centroid/IoU tracking có vấn đề:

  • Khi 2 người đi ngang qua nhau → ID swap → false alert "left zone"
  • Khi occlusion (che khuất) → mất track → re-enter → miss alert
  • Foot point (đáy bbox) không chính xác nếu người đang ngồi/cúi
  • Cần tuning IoU threshold khá nhiều
Giải pháp: Dùng BYTETrack hoặc BoT-SORT (tracker by detection) thay vì centroid/IoU thủ công. Hoặc dùng DeepSORT với appearance feature. Độ chính xác 65% là chấp nhận được nên tracking đơn giản có thể đủ.

🟢 Vấn đề 6: Timing & frame pacing giữa 2 camera

Nếu CAM1 chạy YOLOv8 (~20ms/frame) và CAM2 chạy YOLO-Pose (~35ms/frame), nhưng cả 2 USB camera đều cho frame ở 30fps (33ms/frame):

  • CAM1 kịp xử lý mọi frame, dư tài nguyên
  • CAM2 không kịp → phải drop frame → latency tăng
  • Nếu 2 QThread cùng đọc USB → USB bandwidth có thể không đủ
  • Xử lý occupancy (khi có nhiều người trong frame) làm latency không ổn định
Giải pháp: Frame budget — mỗi thread có max processing time, nếu vượt quá thì skip frame tiếp theo. Dùng queue bounded size để tránh memory leak.
