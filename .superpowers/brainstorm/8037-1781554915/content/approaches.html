3 Approaches — Kiến trúc GPU Machine

YOLOv8n multi-class, ZeroMQ transport, Hybrid PyQt + FastAPI

Approach A: 2 QThreads độc lập ⭐ (Khuyên dùng)

GPU Machine — Process duy nhất: Main Thread (PyQt UI + FastAPI) ├── QThread CAM1: │ ZMQ SUB cam1 → YOLOv8n(inst1) → ROI check → overlay → emit frame ├── QThread CAM2: │ ZMQ SUB cam2 → YOLOv8n(inst2) → PPE check → overlay → emit frame └── FastAPI web (async, optional): đọc frame từ shared queue → serve
✅ Ưu điểm:
  • Đơn giản nhất — 1 process, 3 threads, dễ hiểu, dễ debug
  • 2 pipeline hoàn toàn độc lập — CAM1 skip frame không ảnh hưởng CAM2
  • Mỗi QThread self-paced — frame budget per thread
  • ZeroMQ auto-reconnect khi edge mất kết nối
⚠️ Nhược điểm:
  • 2 instance YOLOv8n → ~12MB VRAM (không đáng kể với GPU)
  • GIL vẫn ảnh hưởng Python code hậu xử lý

Approach B: Pipeline Queue + Shared Model

GPU Machine: Main Thread (UI) ├── QThread CAM1_Recv: ZMQ → frame_queue ├── QThread CAM2_Recv: ZMQ → frame_queue ├── InferencePool (2 workers, shared model): │ frame_queue → inference → result_queue └── Main: result_queue → overlay → render + FastAPI
✅ Ưu điểm:
  • 1 model instance, ít VRAM hơn
  • Có thể điều tiết frame rate qua queue size
❌ Nhược điểm:
  • Queue synchronization phức tạp, dễ deadlock
  • CAM1 latency cao → CAM2 bị ảnh hưởng (shared pool)
  • Khó debug hơn nhiều
  • Với YOLOv8n 6MB, 2 instance không đáng để đánh đổi

Approach C: 2 Processes (Inference + UI riêng)

GPU Machine: Process 1 (Inference): ├── ZMQ SUB cam1 → YOLOv8n → kết quả → ZMQ PUB result └── ZMQ SUB cam2 → YOLOv8n → kết quả → ZMQ PUB result Process 2 (PyQt UI + FastAPI): ├── ZMQ SUB result → overlay → render └── FastAPI web
✅ Ưu điểm:
  • Tách biệt UI và inference — restart UI không mất inference
  • Không bị GIL (inference process CPU-bound)
❌ Nhược điểm:
  • Giống architecture cũ (multiprocessing) — lại có IPC
  • ZMQ IPC overhead (copy frame lần 2)
  • Phức tạp không cần thiết cho project này

Khuyến nghị: Approach A

Lý do: Với YOLOv8n (6MB ONNX, ~20ms/frame trên GPU), 2 instance chỉ tốn ~12MB VRAM — không đáng kể. Lợi ích của sự đơn giản và độc lập lớn hơn nhiều so với tiết kiệm model instance.

Cụ thể:

  • GPU hiện tại: giả sử 4GB+ VRAM → 12MB là ~0.3% VRAM
  • Mỗi QThread có frame budget riêng (skip khi quá tải)
  • Không cần queue, không cần lock, không cần IPC
  • Signal/slot an toàn từ QThread → main thread để render
